# Wiki Schema

この schema は、LLM Wiki のページ種別、配置、必須要素、概念関係の表し方を定義する。新しいページを作るとき、また lint するときの基準として使う。

## ページ種別

### index page

- 配置: `wiki/index.md`
- 役割: wiki 全体の入口。map page へのリンクだけを持ち、concept page や source page へは各 map page から降りる。
- 必須要素:
  - `## 概念マップ`
- 期待するリンク:
  - すべての map page
  - concept page、source page、log page、schema page へは直接リンクしない。

### log page

- 配置: `wiki/log.md`
- 役割: 取り込み、質問対応、点検、整理、運用変更の追記専用ログ。
- リンク方針: log page からは Obsidian 内部リンクを張らない。ページ名やファイルパスは通常テキストまたはコード表記で残す。
- 必須形式:
  - `## [YYYY-MM-DD] 種別 | タイトル`
- 種別:
  - `取り込み`
  - `質問対応`
  - `点検`
  - `整理`
  - `ops`

### glossary page

- 配置: `wiki/glossary.md`
- 役割: 専門用語の英日対訳、採用訳、確認状況、根拠、確認日を管理する。
- リンク方針: glossary page からは Obsidian 内部リンクを張らない。根拠欄には標準訳確認に使った外部資料または訳語判断の短い理由を残し、リンクにならない wiki 内ファイルパスは記入しない。
- 必須セクション:
  - `## 運用ルール`
  - `## 対訳`
- 必須列:
  - `英語`
  - `採用する日本語`
  - `状態`
  - `分野`
  - `根拠`
  - `備考`
  - `確認日`
- 状態:
  - `確認済み`: 信頼できる日本語資料で標準的な訳語を確認した。
  - `暫定`: 構成語や限定的な資料から採用したが、標準訳としては追加確認が必要。
  - `未確立`: 標準的な日本語訳が見つからず、英語表記を維持する。

### source page

- 配置: `wiki/sources/<source-slug>.md`
- 役割: 一次資料または安定した Web source のメタデータ、要約、要点、関連ページを保持する。
- 必須セクション:
  - `## メタデータ`
  - `## 要約`
  - `## 要点`
  - `## 関連ページ`
- 必須メタデータ:
  - `元URL`
  - `取得日`
  - `バージョン`
  - `参照方法`
- 期待するリンク:
  - 関連する concept page
  - snapshot を保存した場合は `raw/` 内のファイルパス

### concept page

- 配置: `wiki/concepts/<concept-slug>.md`
- 役割: 複数の資料や質問から再利用できる概念、主張、関係を統合する。
- 推奨セクション:
  - `## 概要`
  - `## 要点`
  - `## 関連ページ` は、本文や要点では説明しきれない補助的な関係を短く整理できる場合だけ置く。
- 期待するリンク:
  - 根拠となる source page
  - 上位概念、下位概念、特殊例、隣接概念
- 注意:
  - source page の要約を貼り直すだけにしない。
  - 矛盾や更新差分がある場合は、本文中に明記する。
  - 根拠資料や概念関係へのリンクは、原則として本文、要点、関係説明の中に置き、末尾のリンク集だけに退避しない。

### map page

- 配置: `wiki/maps/<topic-slug>.md`
- 役割: 概念群のまとまりと関係を整理する。読む順序ではなく、概念構造を表す。
- 必須セクション:
  - `## 中心概念`
  - `## 概念構造`
  - `## 関係`
  - `## 根拠資料`
- 任意セクション:
  - `## 隣接トピック`
  - `## 関連ページ`
- 期待するリンク:
  - 中心となる concept page
  - 概念構造に含まれる concept page
  - 根拠となる source page
  - 隣接する map page

## 概念関係の語彙

map page と concept page では、概念間の関係をできるだけ次の語彙で明示する。

- 上位概念: ある概念を包む、より一般的な概念。
- 下位概念: 上位概念を具体化した概念。
- 特殊例: 条件を追加したり、特定の場合に限定した例。
- 道具: ある概念を解析、分類、構成するために使う概念。
- 交差: 2つ以上の概念領域が重なる位置にある概念。
- 隣接領域: 直接の上位下位関係ではないが、問いや手法を共有する領域。
- 根拠資料: その概念整理を支える source page または raw source。

## リンク方針

- 内部リンクは、根拠、上位概念、下位概念、特殊例、道具、交差、隣接領域など、概念整理に必要な関係を表すために使う。
- 目次、source page、concept page、map page のリンクは、単なる出口リンク集ではなく、本文やページ種別の役割に沿った関係を持つものに絞る。
- concept page の `## 関連ページ` や map page の `## 隣接トピック` は任意の補助節であり、意味ある関係を説明できない場合は置かない。
- source page の `## 関連ページ` は必須節だが、資料内容と直接対応する concept page や map page に絞る。資料名が同じ、後で読むかもしれない、出口として便利、といった理由だけでは追加しない。
- `## 関連ページ` が内部リンクだけの箇条書きになり、各リンクの関係が本文や節名から説明できない場合は、リンクを本文中の根拠・概念関係へ移すか、節ごと削除する。
- lint では、重複リンク、習慣的に置いただけのリンク、ページの主題と弱くしか関係しないリンクを削る。

## 本文の対象

- concept page、source page、map page の本文は、概念、資料内容、概念間の関係を整理する。
- concept page、source page、map page には、wiki 固有の事情、現在の収録状況、作業状態、運用判断を書かない。
- `この wiki では`、`本 wiki では`、`現時点でこの wiki にある`、`既存ページ`、`source page に残す`、`concept page に戻す` のような自己言及は、概念関係や資料内容へ言い換える。
- wiki 固有の運用方針、作業理由、点検結果、未整理の状態は、`AGENTS.md`、`wiki/schema.md`、`wiki/log.md`、または glossary page の運用ルールに記録する。
- 資料が限定的であることは、その資料の性質として書く。たとえば「根拠資料は大会発表要旨1件であるため」や「要旨本文を十分に確認できていないため」のように、wiki の収録状況ではなく根拠の性質を主語にする。

## 専門用語の表記

- 専門用語は、可能な限り標準的な日本語訳で書く。
- 新しい専門用語を書く前に `wiki/glossary.md` を確認する。
- 対訳表にない専門用語は、信頼できる日本語資料で標準訳を調べてから使う。
- J-GLOBAL の機械翻訳や同種の自動翻訳ページは、訳語採用の根拠にしない。
- 標準訳が確認できた場合は `wiki/glossary.md` に追加し、本文では日本語訳を優先する。
- 標準訳が見つからない場合は、英語を維持し、対訳表に `未確立` として記録する。
- 初出、曖昧な訳語、複数訳がある用語では、日本語訳の後に英語を丸括弧で併記してよい。

## 数式表記

- Markdown/Obsidian 互換を優先し、数式は LaTeX 形式で書く。
- 文中の短い数式は `$...$` のインライン数式にする。
- 独立して読ませたい数式、定義式、目的関数、制約付き最適化、複数行の式は `$$...$$` の表示数式にする。
- 複数行の式や等号をそろえたい式では、表示数式の中で `aligned` を使う。
- 日本語本文では、可能な限り標準的な日本語訳を使い、記号や変数だけを LaTeX にする。
- 新しい記号を導入した直後には、必要に応じて「ここで $A$ は隣接行列、$d$ は次数」のように記号の意味を説明する。
- 長い式を無理に1行へ押し込まず、読みにくい場合は意味のまとまりで改行する。
- 単純な変数名や関数名であっても、数式として意味を持つ場合はバッククォートではなく `$...$` を使う。コード、ファイル名、URL、コマンドは引き続きバッククォートを使う。

例:

```markdown
$d$-正則グラフ $G$ の隣接行列を $A$ とし、その固有値を

$$
d = \lambda_1 \ge \lambda_2 \ge \cdots \ge \lambda_n
$$

と書く。第二固有値 $\lambda_2$ が小さいほど、グラフはよく拡大する。
```

整列が必要な場合:

```markdown
$$
\begin{aligned}
h(G) &= \min_{S \subseteq V,\ |S| \le |V|/2}
       \frac{|\partial S|}{|S|} \\
\lambda(G) &= \max \{ |\lambda_2|, |\lambda_n| \}
\end{aligned}
$$
```

## Lint 観点

- `wiki/index.md` は、実在する map page だけを漏れなく索引し、concept page、source page、log page、schema page へ直接リンクしない。
- Obsidian 形式の内部リンクは実在する wiki page を指す。
- 内部リンクは意味ある概念関係または根拠関係を表し、単なる出口リンクになっていない。
- concept page 末尾に、関係説明のない内部リンクだけの `関連ページ` または `関連` 節が残っていない。
- source page は必須メタデータと必須セクションを持つ。
- map page は必須セクションを持ち、読む順序ではなく概念関係を表す。
- concept page は少なくとも1つの source page または map page とつながる。
- concept page、source page、map page の本文は、wiki 固有の事情や現在の状態ではなく、概念、資料内容、概念関係を述べている。
- glossary page は運用ルール、対訳表、必須列を持つ。
- glossary page は Obsidian 内部リンクを持たない。
- log page は Obsidian 内部リンクを持たない。
- 専門用語の英語表記が残る場合は、固有名詞、論文タイトル、URL、コード、引用、略語、未確立語、初出併記のいずれかとして説明できる状態にする。
- 数式は `$...$` または `$$...$$` の LaTeX 形式で表記し、コード表記と混同しない。
- `wiki/log.md` には、schema 変更、構造化、lint の結果を追記する。
